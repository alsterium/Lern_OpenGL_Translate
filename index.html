Advanced GLSL

Advanced-OpenGL/Advanced-GLSL

この章では、シーンのビジュアルクオリティーを大幅に向上させる、超先進的でクールな新機能を紹介することはできません。この章では、多かれ少なかれ、GLSL の興味深い側面と、将来の努力に役立つかもしれない素敵なトリックを紹介します。基本的には、GLSLと組み合わせてOpenGLアプリケーションを作成する際に、あなたの生活をより簡単にしてくれるかもしれない、いくつかの良い知識と機能があります。

興味深い組み込み変数、シェーダの入出力を整理するための新しい方法、ユニフォームバッファオブジェクトと呼ばれる非常に便利なツールについて説明します。

GLSL's built-in variables

シェーダは非常にパイプライン化されており、現在のシェーダ以外のソースからのデータが必要な場合は、データを渡す必要があります。頂点アトリビュート、ユニフォーム、サンプラーを使って、これを行うことを学びました。しかし、GLSL で定義されている gl_がプレフィックスとなっている変数がいくつかあり、データを収集したり書き込んだりするための特別な手段を提供してくれます。頂点シェーダの出力ベクトルであるgl_Positionと、フラグメントシェーダのgl_FragCoordです。

We'll discuss a few interesting built-in input and output variables that are built-in in GLSL and explain how they may benefit us. Note that we won't discuss all built-in variables that exist in GLSL so if you want to see all built-in variables you can check OpenGL's wiki.

Vertex shader variables

We've already seen gl_Position which is the clip-space output position vector of the vertex shader. Setting gl_Position in the vertex shader is a strict requirement if you want to render anything on the screen. Nothing we haven't seen before.

gl_PointSize

One of the render primitives we're able to choose from is GL_POINTS in which case each single vertex is a primitive and rendered as a point. It is possible to set the size of the points being rendered via OpenGL's glPointSize function, but we can also influence this value in the vertex shader.

One output variable defined by GLSL is called gl_PointSize that is a float variable where you can set the point's width and height in pixels. By setting the point's size in the vertex shader we get per-vertex control over this point's dimensions.

Influencing the point sizes in the vertex shader is disabled by default, but if you want to enable this you'll have to enable OpenGL's GL_PROGRAM_POINT_SIZE: 


glEnable(GL_PROGRAM_POINT_SIZE);

A simple example of influencing point sizes is by setting the point size equal to the clip-space position's z value which is equal to the vertex's distance to the viewer. The point size should then increase the further we are from the vertices as the viewer. 


void main()
{
    gl_Position = projection * view * model * vec4(aPos, 1.0);    
    gl_PointSize = gl_Position.z;    
}

The result is that the points we've drawn are rendered larger the more we move away from them:

You can imagine that varying the point size per vertex is interesting for techniques like particle generation.

gl_VertexID

The gl_Position and gl_PointSize are output variables since their value is read as output from the vertex shader; we can influence the result by writing to them. The vertex shader also gives us an interesting input variable, that we can only read from, called gl_VertexID.

The integer variable gl_VertexID holds the current ID of the vertex we're drawing. When doing indexed rendering (with glDrawElements) this variable holds the current index of the vertex we're drawing. When drawing without indices (via glDrawArrays) this variable holds the number of the currently processed vertex since the start of the render call.

Fragment shader variables

Within the fragment shader we also have access to some interesting variables. GLSL gives us two interesting input variables called gl_FragCoord and gl_FrontFacing.

gl_FragCoord

We've seen the gl_FragCoord a couple of times before during the discussion of depth testing, because the z component of the gl_FragCoord vector is equal to the depth value of that particular fragment. However, we can also use the x and y component of that vector for some interesting effects.

The gl_FragCoord's x and y component are the window- or screen-space coordinates of the fragment, originating from the bottom-left of the window. We specified a render window of 800x600 with glViewport so the screen-space coordinates of the fragment will have x values between 0 and 800, and y values between 0 and 600.

Using the fragment shader we could calculate a different color value based on the screen coordinate of the fragment. A common usage for the gl_FragCoord variable is for comparing visual output of different fragment calculations, as usually seen in tech demos. We could for example split the screen in two by rendering one output to the left side of the window and another output to the right side of the window. An example fragment shader that outputs a different color based on the fragment's screen coordinates is given below: 


void main()
{             
    if(gl_FragCoord.x < 400)
        FragColor = vec4(1.0, 0.0, 0.0, 1.0);
    else
        FragColor = vec4(0.0, 1.0, 0.0, 1.0);        
}

Because the width of the window is equal to 800, whenever a pixel's x-coordinate is less than 400 it must be at the left side of the window and we'll give that fragment a different color.

We can now calculate two completely different fragment shader results and display each of them on a different side of the window. This is great for testing out different lighting techniques for example.

gl_FrontFacing

Another interesting input variable in the fragment shader is the gl_FrontFacing variable. In the face culling chapter we mentioned that OpenGL is able to figure out if a face is a front or back face due to the winding order of the vertices. The gl_FrontFacing variable tells us if the current fragment is part of a front-facing or a back-facing face. We could, for example, decide to output different colors for all back faces.

The gl_FrontFacing variable is a bool that is true if the fragment is part of a front face and false otherwise. We could create a cube this way with a different texture on the inside than on the outside: 


#version 330 core
out vec4 FragColor;
  
in vec2 TexCoords;

uniform sampler2D frontTexture;
uniform sampler2D backTexture;

void main()
{             
    if(gl_FrontFacing)
        FragColor = texture(frontTexture, TexCoords);
    else
        FragColor = texture(backTexture, TexCoords);
}

If we take a peek inside the container we can now see a different texture being used.

Note that if you enabled face culling you won't be able to see any faces inside the container and using gl_FrontFacing would then be pointless.

gl_FragDepth

入力変数gl_FragCoordは、画面空間座標を読み込んで現在のフラグメントの深度値を取得することができる入力変数ですが、読み取り専用の変数です。フラグメントのスクリーン空間座標に影響を与えることはできませんが、フラグメントの深度値を設定することは可能です。GLSLはgl_FragDepthという出力変数を与えてくれます。これを使って、シェーダ内でフラグメントの深度値を手動で設定することができます。

シェーダの深さの値を設定するには、0.0から1.0の間の任意の値を出力変数に書きます。 


gl_FragDepth = 0.0; // this fragment now has a depth value of 0.0

シェーダがgl_FragDepthに何も書き込まない場合、変数は自動的に gl_FragCoord.zから値を取ります。

しかし、手動で深度値を設定することには大きな欠点があります。それは、フラグメントシェーダでgl_FragDepth に書き込むとすぐに、OpenGLは初期の深度テスト ( 深度テストで議論されているように）を無効にしてしまうからです。これが無効になっているのは、OpenGLがフラグメントシェーダを実行する前に、フラグメントがどのような深さの値を持つかを知ることができないからです。

gl_FragDepth に書き込むことで、このパフォーマンスのペナルティを考慮する必要があります。しかし、OpenGL 4.2からは、フラグメントシェーダの先頭に深度条件付きで gl_FragDepth 変数を再定義することで、両者の間を仲介することができるようになりました。: 


layout (depth_<condition>) out float gl_FragDepth;

このcondition は以下の値を取ることができます。:

Condition Description
any The default value. Early depth testing is disabled.
greater You can only make the depth value larger compared to gl_FragCoord.z.
less You can only make the depth value smaller compared to gl_FragCoord.z.
unchanged If you write to gl_FragDepth, you will write exactly gl_FragCoord.z.

深さの条件として大きいか小さいかを指定することで、OpenGLは、フラグメントの深さの値より大きいか小さい深さの値しか書かないという前提を作ることができます。このようにして、OpenGLは、深度バッファ値が gl_FragCoord.z の反対方向の一部である場合でも、初期の深度テストを行うことができます。.

フラグメントシェーダの深度値を大きくしても、初期の深度テストの一部を残したい場合の例は、以下のフラグメントシェーダに示されています。: 


#version 420 core // note the GLSL version!
out vec4 FragColor;
layout (depth_greater) out float gl_FragDepth;

void main()
{             
    FragColor = vec4(1.0);
    gl_FragDepth = gl_FragCoord.z + 0.1;
}

この機能は、OpenGLバージョン4.2以上でのみ利用可能です。

Interface blocks

これまでのところ、頂点からフラグメントシェーダにデータを送るたびに、いくつかの一致する入出力変数を宣言していました。これらの変数を一つずつ宣言するのが、あるシェーダから別のシェーダにデータを送る最も簡単な方法ですが、アプリケーションが大きくなってくると、いくつかの変数以上のものを送りたいと思うようになるでしょう。

これらの変数を整理するために、GLSLは インターフェースブロック と呼ばれるものを提供しています。このようなインターフェースブロックの宣言は struct の宣言によく似ていますが、ブロックが入力ブロックか出力ブロックかに基づいて in または out キーワードを使って宣言されていることを除いては、このような宣言はできません。 


#version 330 core
layout (location = 0) in vec3 aPos;
layout (location = 1) in vec2 aTexCoords;

uniform mat4 model;
uniform mat4 view;
uniform mat4 projection;

out VS_OUT
{
    vec2 TexCoords;
} vs_out;

void main()
{
    gl_Position = projection * view * model * vec4(aPos, 1.0);    
    vs_out.TexCoords = aTexCoords;
}

今回は、次のシェーダに送りたいすべての出力変数をグループ化する vs_out vs_out というインターフェースブロックを宣言しました。これは些細な例ですが、これがシェーダの入出力を整理するのに役立つことは想像できるでしょう。また、 次のジオメトリシェーダの章 で説明するように、シェーダの入出力を配列にグループ化したい場合にも便利です。

次に、次のシェーダであるフラグメントシェーダの入力インターフェースブロックを宣言する必要があります。 ブロック名 (VS_OUT) はフラグメントシェーダでも同じにしますが、インスタンス名 (頂点シェーダで使用されているvs_out ) は好きな名前にしてください - 入力値を含むフラグメント構造体のvs_out のような名前を混乱させないように。 


#version 330 core
out vec4 FragColor;

in VS_OUT
{
    vec2 TexCoords;
} fs_in;

uniform sampler2D texture;

void main()
{             
    FragColor = texture(texture, fs_in.TexCoords);   
}

両方のインターフェイスブロック名が同じである限り、対応する入力と出力が一致します。これもコードの整理に役立つ便利な機能で、ジオメトリシェーダのような特定のシェーダステージ間で交差するときに便利です。

Uniform buffer objects

私たちは、かなり長い間OpenGLを使ってきて、いくつかのクールなトリックを学んできましたが、いくつかのイライラもありました。例えば、複数のシェーダを使用する場合、私たちは継続的に統一変数を設定しなければなりませんが、そのほとんどは各シェーダに対して全く同じです。

OpenGLは、 uniform buffer objects と呼ばれるツールを提供してくれます。これは、いくつものシェーダプログラムに渡って同じままのglobal 変数のセットを宣言できるようにしてくれます。 を使用する場合、関連するユニフォームを固定 GPU メモリに 一度だけ 設定します。 シェーダごとに固有のユニフォームを手動で設定する必要があります。しかし、ユニフォームバッファオブジェクトを作成して設定するのには少々の作業が必要です。

ユニフォームバッファオブジェクトは他のバッファと同じようにバッファなので、 glGenBuffersで作成し、 GL_UNIFORM_BUFFER バッファターゲットにバインドして、関連するユニフォームデータをすべてバッファに格納することができます。ユニフォームバッファオブジェクトのデータをどのように保存するかについてはルールがあり、それについては後ほど説明します。まず、単純な頂点シェーダを用いて、投影行列 と ビュー行列 をuniform blockに格納します。: 


#version 330 core
layout (location = 0) in vec3 aPos;

layout (std140) uniform Matrices
{
    mat4 projection;
    mat4 view;
};

uniform mat4 model;

void main()
{
    gl_Position = projection * view * model * vec4(aPos, 1.0);
}

ほとんどのサンプルでは、使用しているシェーダごとにフレームごとに投影とビューの一様行列を設定しています。これは、これらの行列を一度だけ格納するだけで済むようになったので、一様バッファオブジェクトが便利な例です。

ここでは、2 つの 4x4 の行列を格納する Matrices と呼ばれる一様ブロックを宣言しています。 統一ブロック内の変数は、ブロック名をプレフィックスとして指定しなくても直接アクセスすることができます。そして、これらの行列の値をOpenGLコードのどこかのバッファに格納し、この統一ブロックを宣言した各シェーダが行列にアクセスできるようにします。

今、 layout (std140) 文が何を意味するのか疑問に思っているかもしれません。これは、現在定義されている uniform block layoutが、その内容のために特定のメモリ レイアウトを使用していることを意味します。

Uniform block layout

一様ブロックの内容はバッファオブジェクトに格納されていますが、これは事実上、グローバル GPU メモリの予約された部分に過ぎません。このメモリはどのようなデータを保持しているかの情報を保持していないので、メモリのどの部分がシェーダ内のどのユニフォーム変数に対応しているかをOpenGLに伝える必要があります。

シェーダ内の次の一様ブロックを想像してみてください。:


layout (std140) uniform ExampleBlock
{
    float value;
    vec3  vector;
    mat4  matrix;
    float values[3];
    bool  boolean;
    int   integer;
};

私たちが知りたいのは、これらの変数のサイズ（バイト単位）とオフセット（ブロックの開始から）です。各要素のサイズは、OpenGLで明確に述べられており、C++のデータ型に直接対応しています; ベクトルと行列は、（大きな）フロートの配列です。 OpenGLが明確に述べていないのは、変数間の間隔 です。 これは、ハードウェアが適切と思うように変数を配置したり、パッドを入れたりすることを可能にします。ハードウェアは、例えば、floatの隣に vec3 を配置することができます。すべてのハードウェアがこれを扱えるわけではなく、float を追加する前に vec3 を 4 つの float の配列にパッドします。 素晴らしい機能ですが、私たちにとっては不便です。

デフォルトでは、 共有 レイアウトと呼ばれる一様なメモリレイアウトを使用します - 共有とは、一度ハードウェアによってオフセットが定義されると、複数のプログラム間で一貫して 共有 されるからです。 共有レイアウトでは、変数の順番がそのままである限り、GLSLは最適化のために一様変数を再配置することができます。各一様変数のオフセットがわからないので、一様バッファを正確に埋める方法がわかりません。 glGetUniformIndicesのような関数を使ってこの情報を問い合わせることができますが、この章ではこのアプローチを取りません。

共有レイアウトを使用することでスペースを節約して最適化することができますが、各ユニフォーム変数のオフセットを問い合わせる必要があるため、多くの作業が必要になります。 しかし、一般的には共有レイアウトを使用せず、 std140 レイアウトを使用するのが一般的です。 std140レイアウトは、一連のルールによって管理されるそれぞれのオフセットを標準化することで、各変数タイプのメモリレイアウトを明示的に 示しています。これが標準化されているので、各変数のオフセットを手動で求めることができます。

各変数は、std140 レイアウト・ルールを使用した統一ブロック内で変数が取るスペース（パディングを含む）に等しい ベース・アラインメント を持っています。 各変数について、 アラインメントされたオフセットを計算します：ブロックの開始位置からの変数のバイトオフセット。変数のアラインメントされたバイトオフセットは、そのベースアラインメントの倍数に 必ず 等しくなければなりません。 これは少し口惜しいですが、すぐにいくつかの例を見て、物事を明確にしましょう。

正確なレイアウトルールは、 OpenGLの統一バッファ仕様を参照してください。もっとも一般的なルールについては以下に記します。 GLSLの int,、float 、 bool などの各変数型は、4バイトの各エンティティをNとして表現した4バイトの量として定義されています。

Type Layout rule
Scalar e.g. int or bool Each scalar has a base alignment of N.
Vector Either 2N or 4N. This means that a vec3 has a base alignment of 4N.
Array of scalars or vectors Each element has a base alignment equal to that of a vec4.
Matrices Stored as a large array of column vectors, where each of those vectors has a base alignment of vec4.
Struct Equal to the computed size of its elements according to the previous rules, but padded to a multiple of the size of a vec4.

ほとんどのOpenGLの仕様と同様に、例を示すと理解しやすくなります。先ほど紹介した ExampleBlock と呼ばれる均一なブロックを取り、そのメンバーのそれぞれの整列したオフセットをstd140レイアウトを使って計算しています。: 


layout (std140) uniform ExampleBlock
{
                     // base alignment  // aligned offset
    float value;     // 4               // 0 
    vec3 vector;     // 16              // 16  (offset must be multiple of 16 so 4->16)
    mat4 matrix;     // 16              // 32  (column 0)
                     // 16              // 48  (column 1)
                     // 16              // 64  (column 2)
                     // 16              // 80  (column 3)
    float values[3]; // 16              // 96  (values[0])
                     // 16              // 112 (values[1])
                     // 16              // 128 (values[2])
    bool boolean;    // 4               // 144
    int integer;     // 4               // 148
};

練習として、自分でオフセット値を計算して、この表と比較してみてください。これらの計算されたオフセット値を使って、std140レイアウトのルールに基づいて、 glBufferSubDataのような関数を使って適切なオフセットでバッファにデータを充填することができます。最も効率的ではありませんが、std140レイアウトは、この統一されたブロックを宣言した各プログラムでメモリレイアウトが同じであることを保証してくれます。

統一ブロックの定義にステートメント layout (std140) を追加することで、この統一ブロックがstd140レイアウトを使用することをOpenGLに伝えます。バッファを埋める前に各オフセットを照会する必要があるレイアウトが他にも2つあります。 すでに shared レイアウト を見てきましたが、他の残りのレイアウトは packedされています。 packed されたレイアウトを使用すると、シェーダごとに異なる可能性のあるユニフォームブロックから離れたユニフォーム変数をコンパイラが最適化することができるため、プログラム間でレイアウトが同じままであるという保証はありません（共有ではありません）。

Using uniform buffers

統一ブロックを定義し、そのメモリレイアウトを指定しましたが、実際にどのように使用するかはまだ説明していません。

まず、ユニフォームバッファオブジェクトを作成する必要がありますが、これはお馴染みの glGenBuffersで行います。 バッファオブジェクトを作成したら、それを GL_UNIFORM_BUFFER ターゲットにバインドし、 glBufferDataを呼び出して十分なメモリを確保します。 


unsigned int uboExampleBlock;
glGenBuffers(1, &uboExampleBlock);
glBindBuffer(GL_UNIFORM_BUFFER, uboExampleBlock);
glBufferData(GL_UNIFORM_BUFFER, 152, NULL, GL_STATIC_DRAW); // allocate 152 bytes of memory
glBindBuffer(GL_UNIFORM_BUFFER, 0);

これで、バッファにデータを更新したり挿入したりしたいときはいつでも、uboExampleBlock にバインドし、 glBufferSubData tを使ってメモリを更新することができるようになりました。 この一様なバッファを一度だけ更新するだけで、このバッファを使用するすべてのシェーダは、その更新されたデータを使用するようになりました。しかし、OpenGLはどの統一バッファがどの統一ブロックに対応しているかをどのようにして知っているのでしょうか？

OpenGLのコンテキストでは、ユニフォームバッファをリンクすることができるいくつかの バインディングポイントが定義されています。 一旦ユニフォームバッファを作成したら、それらのバインディングポイントの一つにリンクし、シェーダ内のユニフォームブロックも同じバインディングポイントにリンクし、効果的にそれらを一緒にリンクします。次の図はこれを示しています。:

ご覧のように、複数のユニフォームバッファを異なるバインディングポイントにバインドすることができます。シェーダ A とシェーダ B は両方とも同じバインディングポイント 0, にリンクされたユニフォームブロックを持っているので、それらのユニフォームブロックはuboMatricesにある同じユニフォームデータを共有しています。

シェーダのユニフォームブロックを特定のバインディングポイントに設定するには、プログラムオブジェクト、ユニフォームブロックインデックス、リンク先のバインディングポイントを受け取る glUniformBlockBinding を呼び出します。 uniform block index iは、シェーダ内で定義されたユニフォームブロックの位置インデックスです。これは、プログラムオブジェクトとユニフォームブロックの名前を受け付ける glGetUniformBlockIndex を呼び出すことで取得できます。次のように、図からバインディングポイント2にLights のユニフォームブロックを設定することができます。: 


unsigned int lights_index = glGetUniformBlockIndex(shaderA.ID, "Lights");   
glUniformBlockBinding(shaderA.ID, lights_index, 2);

この処理を各シェーダごと に繰り返す必要があることに注意してください。

OpenGLバージョン4.2以降では、別のレイアウト指定子を追加することで、シェーダ内にユニフォームブロックのバインディングポイントを明示的に格納することも可能で、 glGetUniformBlockIndex と glUniformBlockBindingの呼び出しを省くことができます。次のコードは、 Lights ユニフォームブロックのバインディングポイントを明示的に設定しています。: 

layout(std140, binding = 2) uniform Lights { ... };

次に、ユニフォームバッファオブジェクトを同じバインディングポイントにバインドする必要があり、これは glBindBufferBase または glBindBufferRangeのいずれかを使用して実現できます。 


glBindBufferBase(GL_UNIFORM_BUFFER, 2, uboExampleBlock); 
// or
glBindBufferRange(GL_UNIFORM_BUFFER, 2, uboExampleBlock, 0, 152);

関数 glBindbufferBase は、ターゲット、バインディングポイントインデックス、一様なバッファオブジェクトを想定しています。この関数は、uboExampleBlock をバインディングポイント 2;にリンクします。 また、追加のオフセットとサイズパラメータを想定した glBindBufferRange を使用することもできます - この方法では、ユニフォームバッファの特定の範囲のみをバインディングポイントにバインドすることができます。 glBindBufferRange を使用すると、複数の異なるユニフォームブロックを一つのユニフォームバッファオブジェクトにリンクさせることができます。

これですべての設定が完了したので、ユニフォームバッファへのデータの追加を開始します。全てのデータをシングルバイト配列として追加したり、 glBufferSubDataを使ってバッファの一部を更新したりすることができます。ユニフォーム変数の boolean を更新するには、以下のようにユニフォームバッファオブジェクトを更新します。 


glBindBuffer(GL_UNIFORM_BUFFER, uboExampleBlock);
int b = true; // bools in GLSL are represented as 4 bytes, so we store it in an integer
glBufferSubData(GL_UNIFORM_BUFFER, 144, 4, &b); 
glBindBuffer(GL_UNIFORM_BUFFER, 0);

そして、一様ブロック内の他の全ての一様変数にも同じ手順が適用されますが、異なる範囲の引数を使用します。

A simple example

それでは、一様なバッファオブジェクトの実例を示してみましょう。これまでのコードサンプルを振り返ってみると、投影行列、ビュー行列、モデル行列の3つの行列を継続的に使用しています。これらのマトリックスのうち、モデルマトリックスだけが頻繁に変更されています。複数のシェーダが同じマトリックスを使用している場合は、一様なバッファオブジェクトを使用した方が良いでしょう。

ここでは、投影行列とビュー行列をMatricesという一様なブロックに格納します。モデルマトリクスはシェーダ間で頻繁に変更されるので、ここにはモデルマトリクスは格納しません。 


#version 330 core
layout (location = 0) in vec3 aPos;

layout (std140) uniform Matrices
{
    mat4 projection;
    mat4 view;
};
uniform mat4 model;

void main()
{
    gl_Position = projection * view * model * vec4(aPos, 1.0);
}

Not much going on here, except that we now use a uniform block with a std140 layout. What we're going to do in our sample application is display 4 cubes where each cube is displayed with a different shader program. Each of the 4 shader programs uses the same vertex shader, but has a unique fragment shader that only outputs a single color that differs per shader.

First, we set the uniform block of the vertex shaders equal to binding point 0. Note that we have to do this for each shader: 


unsigned int uniformBlockIndexRed    = glGetUniformBlockIndex(shaderRed.ID, "Matrices");
unsigned int uniformBlockIndexGreen  = glGetUniformBlockIndex(shaderGreen.ID, "Matrices");
unsigned int uniformBlockIndexBlue   = glGetUniformBlockIndex(shaderBlue.ID, "Matrices");
unsigned int uniformBlockIndexYellow = glGetUniformBlockIndex(shaderYellow.ID, "Matrices");  
  
glUniformBlockBinding(shaderRed.ID,    uniformBlockIndexRed, 0);
glUniformBlockBinding(shaderGreen.ID,  uniformBlockIndexGreen, 0);
glUniformBlockBinding(shaderBlue.ID,   uniformBlockIndexBlue, 0);
glUniformBlockBinding(shaderYellow.ID, uniformBlockIndexYellow, 0);

Next we create the actual uniform buffer object and bind that buffer to binding point 0: 


unsigned int uboMatrices
glGenBuffers(1, &uboMatrices);
  
glBindBuffer(GL_UNIFORM_BUFFER, uboMatrices);
glBufferData(GL_UNIFORM_BUFFER, 2 * sizeof(glm::mat4), NULL, GL_STATIC_DRAW);
glBindBuffer(GL_UNIFORM_BUFFER, 0);
  
glBindBufferRange(GL_UNIFORM_BUFFER, 0, uboMatrices, 0, 2 * sizeof(glm::mat4));

First we allocate enough memory for our buffer which is equal to 2 times the size of glm::mat4. The size of GLM's matrix types correspond directly to mat4 in GLSL. Then we link a specific range of the buffer, in this case the entire buffer, to binding point 0.

Now all that's left to do is fill the buffer. If we keep the field of view value constant of the projection matrix (so no more camera zoom) we only have to update it once in our application - this means we only have to insert this into the buffer only once as well. Because we already allocated enough memory in the buffer object we can use glBufferSubData to store the projection matrix before we enter the render loop: 


glm::mat4 projection = glm::perspective(glm::radians(45.0f), (float)width/(float)height, 0.1f, 100.0f);
glBindBuffer(GL_UNIFORM_BUFFER, uboMatrices);
glBufferSubData(GL_UNIFORM_BUFFER, 0, sizeof(glm::mat4), glm::value_ptr(projection));
glBindBuffer(GL_UNIFORM_BUFFER, 0);

Here we store the first half of the uniform buffer with the projection matrix. Then before we render the objects each frame we update the second half of the buffer with the view matrix: 


glm::mat4 view = camera.GetViewMatrix();	       
glBindBuffer(GL_UNIFORM_BUFFER, uboMatrices);
glBufferSubData(GL_UNIFORM_BUFFER, sizeof(glm::mat4), sizeof(glm::mat4), glm::value_ptr(view));
glBindBuffer(GL_UNIFORM_BUFFER, 0);

And that's it for uniform buffer objects. Each vertex shader that contains a Matrices uniform block will now contain the data stored in uboMatrices. So if we now were to draw 4 cubes using 4 different shaders, their projection and view matrix should be the same: 


glBindVertexArray(cubeVAO);
shaderRed.use();
glm::mat4 model = glm::mat4(1.0f);
model = glm::translate(model, glm::vec3(-0.75f, 0.75f, 0.0f));	// move top-left
shaderRed.setMat4("model", model);
glDrawArrays(GL_TRIANGLES, 0, 36);        
// ... draw Green Cube
// ... draw Blue Cube
// ... draw Yellow Cube

The only uniform we still need to set is the model uniform. Using uniform buffer objects in a scenario like this saves us from quite a few uniform calls per shader. The result looks something like this:

Each of the cubes is moved to one side of the window by translating the model matrix and, thanks to the different fragment shaders, their colors differ per object. This is a relatively simple scenario of where we could use uniform buffer objects, but any large rendering application can have over hundreds of shader programs active which is where uniform buffer objects really start to shine.

You can find the full source code of the uniform example application here.

Uniform buffer objects have several advantages over single uniforms. First, setting a lot of uniforms at once is faster than setting multiple uniforms one at a time. Second, if you want to change the same uniform over several shaders, it is much easier to change a uniform once in a uniform buffer. One last advantage that is not immediately apparent is that you can use a lot more uniforms in shaders using uniform buffer objects. OpenGL has a limit to how much uniform data it can handle which can be queried with GL_MAX_VERTEX_UNIFORM_COMPONENTS. When using uniform buffer objects, this limit is much higher. So whenever you reach a maximum number of uniforms (when doing skeletal animation for example) there's always uniform buffer objects.
